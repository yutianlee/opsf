# v0.3.0 Final Readiness Audit

Date: 2026-05-12.

This audit reviews the current 0.3 line after `v0.3.0-alpha.5` and the
post-release smoke-pin verification. It is documentation only. It does not
change runtime behavior, public wrappers, default dispatch, version metadata,
workflow defaults, or certification scope.

## Decision

v0.3.0 final is release-ready subject to final release checklist execution.

No alpha.6 blocker was found. The final release still needs a dedicated final
release PR to update release metadata, final release notes, workflow defaults,
and checklist state. That PR must not broaden certification claims beyond the
documented 0.3 scope.

## Release-Claim Guardrails

[`release_claims.md`](release_claims.md) and
`tests/test_release_claims.py` keep conservative release wording for the 0.x
line. They allow narrow wording for:

- direct Arb primitive and finite-product scopes;
- formula-backed alpha scopes where documented;
- experimental formula-layer parabolic-cylinder scopes;
- explicit positive-real custom methods for finite real `x >= 20`; and
- explicit `method="certified_auto"` as a certified-mode selector only.

They continue to forbid language implying package-wide completion, all-input
gamma-family coverage, complex-domain Stirling coverage, reflection-formula
support, near-pole support, gamma-ratio or beta asymptotic certification,
automatic custom-method selection, or non-experimental parabolic-cylinder
promotion.

## Custom-Method Scope

| Wrapper | Explicit method | Domain | Runtime method | Certificate scope | Default dispatch |
| --- | --- | --- | --- | --- | --- |
| `loggamma` | `method="stirling"` | finite real `x >= 20` | `stirling_loggamma` | `stirling_loggamma_positive_real` | unchanged |
| `loggamma` | `method="stirling_shifted"` | finite real `x >= 20` | `stirling_shifted_loggamma` | `stirling_loggamma_positive_real` | unchanged |
| `loggamma` | `method="certified_auto"` | explicit selector over direct Arb and positive-real Stirling paths | selected method's runtime method | selected method's scope | unchanged |
| `gamma` | `method="stirling_exp"` | finite real `x >= 20` | `stirling_exp_gamma` | `gamma_positive_real_stirling_exp` | unchanged |
| `rgamma` | `method="stirling_recip"` | finite real `x >= 20` | `stirling_recip_rgamma` | `rgamma_positive_real_stirling_recip` | unchanged |

The custom-method certificate level is `custom_asymptotic_bound`, and the audit
status is `theorem_documented`. These methods do not add complex gamma or
reciprocal-gamma paths, reflection formulas, near-pole behavior, gamma-ratio
asymptotics, beta asymptotics, or parabolic-cylinder promotion.

## Default Dispatch Behavior

Default dispatch remains conservative:

- `method=None` and `method="auto"` keep the previous automatic path.
- Default certified `loggamma` remains direct Arb.
- Default certified `gamma` remains direct Arb.
- Default certified `rgamma` remains direct Arb.
- `method="certified_auto"` remains explicit and does not become the omitted
  method or `method="auto"` behavior.
- `method="stirling_exp"` and `method="stirling_recip"` remain explicit
  certified-mode methods only.

The post-smoke-pin run `25711747679` passed all nine jobs with

## MCP Smoke Coverage

The MCP server remains an experimental interface and does not expand the
mathematical support surface. The current smoke checks import the MCP server,
build the tool registry, and exercise representative wrappers. They also
verify explicit custom-method payloads for:

- `special_loggamma(..., method="stirling")`
- `special_loggamma(..., method="stirling_shifted")`
- `special_loggamma(..., method="certified_auto")`
- `special_gamma(..., method="stirling_exp")`
- `special_rgamma(..., method="stirling_recip")`

The `special_rgamma` checks include both `dps=50` and `dps=100` and assert the
runtime method, backend, selected method, certificate scope, certificate level,
audit status, underlying loggamma method, and absolute error-bound presence.

## Parabolic-Cylinder Status

The parabolic-cylinder wrappers remain an `experimental_formula` surface:

- `pcfu`
- `pcfd`
- `pbdv`
- `pcfv`
- `pcfw`

The formula audit remains intentionally separate from the 0.3 custom Stirling
work. Cutting v0.3.0 final must not promote these wrappers beyond the
documented experimental formula-layer status.

## Release Checklist Implications

[`release_checklist.md`](release_checklist.md) contains enough generic final
release instructions to cut a non-prerelease release safely: normal GitHub
release status, tag/package parity, build checks, `twine check`, TestPyPI
decision, real PyPI publication, and smoke verification.

However, the final release PR should add a dedicated `v0.3.0` checklist section
to make the 0.3-specific scope harder to misstate. Proposed additions:

- `pyproject.toml` version is `0.3.0`.
- `CITATION.cff` version is `0.3.0` and `date-released` matches the release
  date.
- `CHANGELOG.md` records `0.3.0`.
- `docs/release-0.3.0.md` is reviewed and updated from planning notes to final
  release notes where appropriate.
- Publish workflow defaults point to `v0.3.0`.
- `pypi-smoke.yml` remains pinned to `0.3.0a5` until `certsf 0.3.0` is
  published and smoke verification passes.
- GitHub release for `v0.3.0` is a normal release, not a prerelease.
- Stage to TestPyPI under `release_policy.md` because this is the first
  non-prerelease release of the 0.3 line, unless the final release-planning PR
  explicitly documents why staging is skipped.
- Publish the real PyPI release through the protected `publish-pypi` workflow
  from the GitHub release.
- Run `pypi-smoke` against `0.3.0` after real PyPI publication.
- No `src/` runtime implementation changes are included in the final-release
  PR.
- No public-wrapper changes are included.
- Default certified `loggamma`, `gamma`, and `rgamma` remain direct Arb.
- `method=None` and `method="auto"` remain unchanged.
- Custom 0.3 methods remain explicit only.
- Parabolic-cylinder wrappers remain `experimental_formula`.
- Release copy does not broaden certification claims.

## Required Final-Release Sequence

1. Open a final-release PR from current `main`.
2. Update only release metadata, final release notes, docs, checklist entries,
   and workflow defaults needed for `v0.3.0`.
3. Set `pyproject.toml` to `version = "0.3.0"`.
4. Set `CITATION.cff` to version `0.3.0` and the final release date.
5. Update `CHANGELOG.md` with a `0.3.0` final section and a fresh
   `Unreleased` section.
6. Update publish workflow default refs to `v0.3.0`.
7. Keep `pypi-smoke.yml` pinned to `0.3.0a5` until real PyPI publication and
   final smoke verification pass.
8. Run final PR validation:

   ```powershell
   python scripts/check_release_version.py v0.3.0
   python -m ruff check .
   python -m mypy
   python -m pyright src
   python -m pytest
   python -m build
   python -m twine check dist/*
   ```

9. After merge, tag the final `main` commit as `v0.3.0`.
10. Stage to TestPyPI with `publish-testpypi.yml` using `ref=v0.3.0` and
    `confirm=publish-testpypi`, unless the final release PR explicitly records
    a skip decision under `release_policy.md`.
11. Verify the TestPyPI package install enough to catch packaging and metadata
    issues.
12. Create a normal GitHub release for `v0.3.0`, not a prerelease.
13. Confirm the protected `publish-pypi` workflow publishes `certsf 0.3.0`.
14. Verify real PyPI clean installs, including certified and MCP/certified
    extras.
15. Run manual `pypi-smoke` with `version=0.3.0` and confirm all base,
    certified, and MCP-certified jobs pass on Python 3.10, 3.11, and 3.12.
16. Open a post-release verification PR recording tag, release URL, PyPI URL,
    workflow runs, source commit, artifact hashes, smoke results, and any PyPI
    propagation delay.
17. After real PyPI smoke verification passes, open a separate smoke-pin PR to
    advance `pypi-smoke.yml` to `0.3.0`.
