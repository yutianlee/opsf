# v0.4.0 Positive-Real Method Closeout

This document records the v0.4.0 positive-real gamma-ratio and beta method
implementation closeout after the completed v0.3.0 final release. It is a
development closeout note only: it does not authorize a release, bump the
package version, change workflow files, publish artifacts, or promote any
explicit method into default dispatch.

## Implemented Explicit Methods

The v0.4.0 positive-real method plan has landed the three explicit certified
methods built from the existing certified positive-real `loggamma` machinery:

| Method | Certificate scope | Implementation PR |
| --- | --- | --- |
| `loggamma_ratio(a, b, mode="certified", method="stirling_diff", dps=...)` | `loggamma_ratio_positive_real_stirling_diff` | PR #122 |
| `gamma_ratio(a, b, mode="certified", method="stirling_ratio", dps=...)` | `gamma_ratio_positive_real_stirling_ratio` | PR #123 |
| `beta(a, b, mode="certified", method="stirling_beta", dps=...)` | `beta_positive_real_stirling_beta` | PR #124 |

All three methods are restricted to finite real `a >= 20` and finite real
`b >= 20` for the two-argument methods. The one-argument positive-real
`loggamma`, `gamma`, and `rgamma` custom methods remain restricted to finite
real `x >= 20`.

## Dispatch Guardrails

The v0.4.0 method work preserves the existing default certified Arb paths.
The explicit methods are selected only when their exact method names are
requested in certified mode.

These dispatch behaviors remain unchanged:

- omitted `method` behavior;
- `method="auto"` behavior;
- default certified dispatch;
- `method="arb"` direct Arb selection; and
- existing direct Arb paths for `loggamma_ratio`, `gamma_ratio`, and `beta`.

## Exclusions

The closeout does not add or claim:

- complex gamma-ratio Stirling certification;
- complex beta Stirling certification;
- reflection-formula certification;
- near-pole behavior;
- simultaneous-pole limiting values;
- default-dispatch promotion;
- parabolic-cylinder promotion;
- new public wrappers;
- workflow changes;
- version changes; or
- broad package-wide certification.

## Release Boundary

Any v0.4.0 release remains a separate action. Per release policy, a release
requires explicit user authorization naming the repository, tag, package
version, GitHub release type, PyPI/TestPyPI choices, and post-release smoke
plan. Without that authorization, the repository remains in development state
with the explicit methods documented and guarded by tests.
